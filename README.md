# llm-wiki-anatomy

> 유행처럼 펼쳐지는 LLM Wiki에 대한 다양한 관점과 본질을 꿰뚫는 레포.

## 당신이 쫓고 있는 것

카파시의 gist를 읽었고, 유튜브에서 옵시디언 vault에 에이전트를 물리는
영상을 봤고, 그래프뷰가 별자리처럼 펼쳐지는 화면에 감탄했을 것이다.
그리고 지금 옵시디언을 설치하고 있을 것이다.

멈춰라. **질문 하나에 답해보라 — 그 위키에 무엇을 검색할 것인가?**

답이 없다면 당신이 만드는 것은 위키가 아니라 요약 무덤이다.
그래프뷰는 예쁘게 나올 것이다. 3주 뒤에 안 열어볼 것도 확실하다.

## 카파시가 말한 것 vs 당신이 쫓는 것

카파시가 말한 것은 **패턴**이다: LLM은 지루해하지 않고, 크로스레퍼런스
갱신을 잊지 않고, 한 번에 15개 파일을 고칠 수 있다 — 그러니 인간이
포기하던 위키 유지보수를 에이전트에게 맡기자는 것.

당신이 쫓는 것은 **도구**다: 옵시디언, 그래프뷰, graphify.
도구는 패턴의 물질화 중 하나일 뿐이다. 증거를 대겠다.

## 구글이 증명해버렸다 — OKF

2026년 6월, 구글 클라우드가 Open Knowledge Format(OKF) v0.1을 발표하며
이렇게 썼다: "LLM-wiki 패턴을 이식 가능하고 상호운용 가능한 **포맷**으로
공식화한다." 스펙의 전부는 마크다운 디렉토리 + frontmatter 필드 `type`
하나 + 파일 간 링크. 옵시디언이라는 단어는 스펙 어디에도 없다.

**본질은 포맷과 구조이지, 도구가 아니라는 뜻이다.**
옵시디언은 그 포맷의 뷰어 중 하나일 뿐이다.

## 본질 세 문장

1. **위키는 저장소가 아니라 검색 계약이다.**
   "어떤 질문을 던질 것인가"를 먼저 정의하고, 그 질문에 답이 나오도록
   지식을 압축·구조화·연결하는 것. 질문이 없으면 오버엔지니어링이다.

2. **모든 지식검색은 3개 층의 조합이다.**
   표현층(어떤 형태로 압축) × 연결층(요약→원문을 어떻게 잇나) ×
   접근층(에이전트가 어떻게 읽나). 옵시디언 방식은 이 조합의 하나다:
   요약 md × 심링크·정션 × 파일읽기. 그래프뷰는 부산물이다.

3. **원문 포인터가 없으면 위키가 아니다.**
   위키에는 요약만 저장한다. 검색된 요약에서 심링크·경로·URL로 원문까지
   에이전트가 도달해야 완성이다. 예쁜 그래프에는 이 층이 자주 빠져 있다.

## 응용 — 같은 본질, 6가지 물질화

같은 검색 계약을 6가지 방식으로 구현할 수 있다. 하나를 골라라.
(이 레포를 클론하고 에이전트에게 CLAUDE.md를 읽히면 메뉴부터 시작한다)

| # | 방식 | 조합 (표현×연결×접근) | 잘 답하는 질문 |
|---|---|---|---|
| [01](recipes/01-wiki-md/) | Wiki md (옵시디언식) | 요약 md × 심링크·정션 × 파일읽기 | 관계 탐색, 그래프뷰 |
| [02](recipes/02-sqlite-db/) | SQLite | 레코드 × 외래키 × SQL | 정확한 조인·필터·집계 |
| [03](recipes/03-mcp-server/) | MCP 서버 | 02 × MCP 툴콜 | 어떤 세션에서든 `/wiki` |
| [04](recipes/04-rag/) | RAG | 벡터 청크 × 메타데이터 × 유사도 | "비슷한 거 어디 있었지" |
| [05](recipes/05-rss-static/) | RSS/정적사이트 | 피드 × URL × HTTP | 구독, 외부 공유 |
| [06](recipes/06-okf/) | OKF 번들 | 요약 md × resource × 표준규약 | 타 조직·타 도구 호환 |

RAG vs Wiki vs DB 논쟁은 가짜 논쟁이다. 대립이 아니라 **같은 온톨로지의
다른 물질화**다. 질문 형태가 조합을 정한다.

## 온톨로지에도 정답은 없다

개념-개체-관계 트리플? 그것도 패턴 중 하나일 뿐이다.
자료와 질문에 따라 계층·패싯·이벤트·최소 구조가 더 맞을 수 있다.
[`ontology/GUIDE.md`](ontology/GUIDE.md)에서 5가지 패턴을 질문 형태로
고르는 법을 다룬다. 단 두 가지만은 패턴과 무관하게 불변이다:
**검색 계약이 먼저**, 그리고 **source_path 필수**.

## LLM wiki가 기존 KMS와 다른 단 하나

RDBMS와 구조는 같다. 다른 점은 복잡한 조인과 키 스키마를 사람이 짜는 게
아니라 **LLM이 의미 기반으로 대신 걸어준다**는 것. 그래서 스키마가
느슨해도 검색이 된다. 단, 무엇을 축으로 삼을지는 사람이 정한다 —
그것이 온톨로지 설계이고, 도구가 대신 못 해주는 유일한 일이다.

트렌드는 도구를 판다. 본질은 질문을 판다.
**옵시디언을 깔기 전에, 검색 문장 세 개를 먼저 써라.**

## 사용법

```bash
git clone https://github.com/cdsassj00/llm-wiki-anatomy.git
cd llm-wiki-anatomy && claude
```
```
CLAUDE.md 읽고 위키 만들어줘. 문서는 D:\lectures 에 있어.
```

에이전트는 위키를 바로 만들지 않는다. 검색 문장 세 개부터 물을 것이다.
그게 이 레포의 존재 이유다.
