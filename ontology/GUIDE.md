# 온톨로지 설계 가이드

## 대전제: 온톨로지에 정답 구조는 없다

개념-개체-관계는 하나의 패턴일 뿐이다. **어떤 문서가 담기고, 무슨 질문을
던지느냐에 따라 구조 자체가 달라진다.** 아래 패턴 중에서 고르거나 섞어라.
에이전트는 사용자에게 이 패턴들을 보여주고 자료 성격에 맞는 것을 함께 고른다.

## 5가지 온톨로지 패턴

| 패턴 | 구조 | 이런 자료·질문에 | 예시 |
|---|---|---|---|
| **① 트리플** (개념-개체-관계) | 그래프. [주어,술어,목적어] | 관계 조인이 핵심 질문일 때 — "X를 가르친 기관과 실습은?" | 강의안 (`ontology.yaml`) |
| **② 계층/분류** (taxonomy) | 트리. 대분류>중분류>소분류 | 자료가 자연스럽게 카테고리로 떨어질 때 — "AI교육 > 바이브코딩 아래 다 보여줘" | 커리큘럼 체계, 법령 |
| **③ 패싯** (faceted) | 독립 태그축들의 조합. 관계 없음 | 축은 명확한데 축끼리 얽히진 않을 때 — "2025년 × 공공 × 컨설팅 필터" | 제안서, 사진, 상품 |
| **④ 시간축/이벤트** | 로그. 사건의 연대기 | "언제 무슨 일이 있었나"가 핵심일 때 | 회의록, 프로젝트 일지, 뉴스 |
| **⑤ 최소/자유** (OKF식) | type 하나 + 자유 링크 | 구조를 미리 못 정하겠을 때. 링크가 쌓이며 구조가 창발 | 잡다한 스크랩, 초기 위키 |

혼합이 보통이다: 제안서 위키라면 패싯(③)으로 필터하고 트리플(①) 관계
한두 개(`document cites 수행실적`)만 얹는 식.

## 패턴 선택법: 역시 질문에서 역산

1. 실제 검색 문장을 3개 이상 쓴다
2. 문장의 형태를 본다:
   - 명사끼리 **얽혀** 있으면("X를 한 곳의 Y") → ① 트리플
   - "~아래", "~계열" 같은 **포함** 표현 → ② 계층
   - "A이면서 B이면서 C인 것" **필터** 표현 → ③ 패싯
   - "언제", "그 다음에" **시간** 표현 → ④ 이벤트
   - 뭘 물을지 아직 모르겠다 → ⑤ 최소로 시작, 나중에 승격
3. 선택한 패턴으로 `my-ontology.yaml`을 작성 (`examples/` 참고)

## 패턴이 달라도 변하지 않는 것 2가지

1. **검색 계약이 먼저다.** 질문 없이 패턴부터 고르면 계륵.
2. **source_path(원문 포인터)는 필수다.** 어떤 패턴이든 위키에는 요약만,
   원문은 경로·링크로 도달. 이게 깨지면 요약 무덤이다.

## 트리플 패턴의 구조 (①을 골랐을 때)

```
   개념:  topic(주제)     theory(이론)     practice(실습)
             ▲               ▲                ▲
   관계:  belongs_to      explains          trains
             │               │                │
   개체:  document(강의안) ──contains──▶ module(모듈)
             │
             ├──delivered_at──▶ org(기관)
             └──source_path───▶ 원문 파일
```

"콘솔"(theory) 검색 → explains 역방향 → module → contains 역방향 →
document → delivered_at → org. 검색 한 번에 기관·이론·실습·주제가 엮인다.

## 체크리스트

- [ ] 검색 문장 3개 이상에서 역산했는가
- [ ] 패턴을 문장 형태에 맞춰 골랐는가 (트리플이 디폴트가 아니다)
- [ ] 축/노드가 3~6개인가 (7개 이상 = 과설계. ⑤로 후퇴 고려)
- [ ] source_path가 있는가
- [ ] 값이 없는 축의 처리를 정했는가 (null 허용 권장)
